## 1. Executive Summary

### Problem Statement

智能客服可能编造优惠、产品参数、退货地址或操作结果。质量负责人缺少一个可批量检测、解释、量化评测并根据误判持续改进的工具。

现有测评样例共 20 条，其中 18 条为人工标注的幻觉，2 条为正常回复。样本量小且类别严重不均衡，产品必须避免用普通准确率制造虚高结论，也必须防止人工答案泄漏到首次检测中。

### Proposed Solution

交付一个仅限本机使用、由 FastAPI 提供唯一 HTTP 服务、Jinja2 服务端渲染、HTMX 局部交互和 ECharts 图表组成的 Web Dashboard。用户可加载内置样例或上传同结构数据，主动确认外部处理后使用真实 LLM 检测幻觉，查看可回指知识库原文的证据，并可按运行选择是否开启人工复审。开启后，用户可逐条确认或结构化修订检测错误；原始预测与人工修订分别保存。系统还能在预测冻结后单独加载官方人工标注，计算漏检、误报和分类指标，并基于误判生成不可执行、不可激活的实验性改进建议。MVP 不自动生成可部署检测器、不激活候选版本，也不提供版本回滚。

### Success Criteria

1. **SC-01 可运行与失败诚实性**：应用可通过 `python -m uvicorn src.api.app:app --reload` 在 loopback 地址启动。自动化测试使用注入的确定性 mock Provider，使内置 20 条样例全部得到结构有效的成功结果；真实 LLM 演示只要求完整记录成功数、失败数、尝试次数和失败原因，不要求非确定性的外部服务达到 100% 成功。任何失败记录必须保留，且不得以跳过失败记录提高指标。
2. **SC-02 检测完整性与可解释性**：100% 的成功结果包含二分类、多标签、可空主类型、可空风险等级、原子声明、关键遗漏、总结、来源和检测器版本；每个原子声明必须通过字符区间精确回指 `system_reply` 原文，`supported`/`contradicted` 的证据引用必须精确回指 `knowledge_base`，`unsupported` 必须明确表示没有支持证据。开启人工复审后，每条成功结果均可确认、修订并保留不可变的原始预测。
3. **SC-03 评测正确性**：自动化指标测试对预置 TP、FP、TN、FN、零分母、ID 不匹配和失败记录夹具的计算结果 100% 正确；所有指标显示覆盖率和明确分母，覆盖率不足时标为“不完整”。
4. **SC-04 隔离与生命周期**：自动化测试证明首次检测请求和检测模块依赖中不存在官方标注、风险参考或人工修订；只有显式冻结的预测快照可加载标注和评测，冻结后任何重试都必须创建新的子运行，原快照哈希保持不变。
5. **SC-05 MVP 安全边界与建议降级**：输入大小、声明数量、LLM 调用次数、累计 token、输出 token、任务时限、用户取消、loopback 访问、同源写操作、HTML/Markdown 转义和外部处理确认均有自动化测试；误判分析只能生成带来源和风险提示的实验性建议 JSON/Markdown，不得创建检测器版本文件、修改活动指针、激活、回滚或执行代码。

## 2. User Experience & Functionality

### User Personas

- **智能客服质量负责人**：批量发现高风险幻觉，定位漏检和误报，并形成需要后续独立验证的改进方向。
- **测评审阅者**：验证工具确实运行、结果有证据、指标无泄漏且局限性说明诚实。

### User Flow

1. 用户打开 Dashboard，加载内置样例或上传同结构 JSON。
2. 用户确认 LLM 配置有效、阅读外部处理提示并主动确认后启动检测。
3. 用户查看总体分布和逐条证据。
4. 若本次运行开启人工复审，用户逐条标记检测正确或有误，并结构化修改错误点。
5. 用户另行加载官方人工标注并运行评测。
6. 用户查看混淆矩阵、漏检、误报和原因分析。
7. 用户选择官方标注或完成复核的人工修订作为单一分析来源，生成不可执行、不可激活的实验性改进建议。
8. 用户导出预测、评测、人工反馈、实验性建议和综合报告。

### User Stories

#### US-01：加载待检测数据

作为质量负责人，我希望一键加载内置样例或上传同结构 JSON，以便演示和复用批量检测能力。

验收标准：

- **AC-01.1** 内置样例使用 `task4_replies.json`，且运行时代码不得修改该文件。
- **AC-01.2** 上传格式支持非空 JSON 数组，必填字段为 `id`、`user_question`、`system_reply`、`knowledge_base`；规范化后的 `id` 必须匹配 `[A-Za-z0-9][A-Za-z0-9._-]{0,127}`，问题和回复去除首尾空格后不得为空，知识库允许为空但页面必须提示闭集判断风险。
- **AC-01.3** 每批最多 20 条；请求体最大 5 MiB；所有记录的问题、回复和知识库字符数合计最多 200,000；`id` 最长 128 字符，问题和回复分别最长 10,000 个 Unicode 字符，知识库最长 50,000 个 Unicode 字符；超限时必须在调用 LLM 前拒绝。
- **AC-01.4** 字段缺失、类型错误、ID 重复、非法控制字符或超限时，页面阻止运行并指出具体字段路径。
- **AC-01.5** MVP 不要求 CSV 和字段映射。

#### US-02：运行幻觉检测

作为质量负责人，我希望使用真实 LLM 批量检测，以便验证基于模型的证据提取与幻觉判断能力。

验收标准：

- **AC-02.1** 运行时只允许符合技术设计中 OpenAI-compatible Chat Completions 契约的真实 LLM Provider，不提供本地规则、离线检测或 mock 模式。
- **AC-02.2** 缺少 `HALLUCINATION_API_KEY`、`HALLUCINATION_BASE_URL` 或 `HALLUCINATION_MODEL`，或 Base URL 不满足“外部地址必须 HTTPS、loopback 可使用 HTTP”时，禁止启动检测并列出无敏感值的配置错误。
- **AC-02.3** 检测只在用户勾选“我确认所选内容将发送到配置的外部 LLM 服务”并主动点击后发起调用；应用启动、页面加载和配置状态检查不得自动调用。
- **AC-02.4** 启动前页面显示记录数、输入字符数、基于“每条最多 10 个声明和 1 次遗漏检查”计算的逻辑调用上界，以及每个检测运行最多 200 次网络请求尝试、250,000 个 Provider 计量 total tokens、单响应最多 2,000 completion tokens 和 30 分钟墙钟时间的边界；这些边界限制风险但不等价于货币费用保证。用户未确认时不得创建运行。
- **AC-02.5** 测试可以注入 mock LLM Provider，但不得通过运行时页面、请求字段或公开 API 启用。
- **AC-02.6** 每条成功预测的 `engine` 必须为 `llm`，并记录 Provider 响应返回的模型名、只读基线检测器版本和配置哈希；不得伪造 LLM 成功结果。
- **AC-02.7** 每条回复最多保留 10 个原子声明；模型返回更多声明时，该条记录以结构错误失败，不得静默截断。每个检测运行的首次执行和冻结前重试累计最多 200 次网络请求尝试；每次请求最多允许 2,000 completion tokens。Provider 计量的累计 `usage.total_tokens` 达到或超过 250,000 后不得继续发起请求，并为剩余记录保存明确失败结果；由于 usage 只能在响应后获得，该计量断路器最多可能被最后一个已发出的请求突破一次，不得表述为严格 token 或货币费用保证。
- **AC-02.8** 页面显示逐条进度、成功数、失败数和尝试次数；单条失败不终止整个批次。
- **AC-02.9** 失败项仅能在运行冻结前单独重试；运行冻结后重试必须创建不继承标注和人工修订的新子运行，原快照不可改变。
- **AC-02.10** MVP 同一应用 worker 同时只执行一个检测或建议生成任务；第二个启动请求返回 `409`，不得排队或并行消耗外部额度。
- **AC-02.11** 用户可取消活动检测任务；检测任务开始 30 分钟后也会触发相同的协作式停止。正在进行的 HTTP 请求最多等待其 60 秒 socket timeout，不强制中断线程；随后不再发起新请求，尚未形成完整结果的当前记录及后续记录保存为 `cancelled` 或 `run_deadline_exceeded`，运行进入 `retryable_partial`。实验性建议任务的墙钟上限为 5 分钟，取消或超时后整项失败且不保存部分建议。

#### US-03：查看可解释结果

作为质量负责人，我希望看到回复中的具体错误声明和知识库证据，以便理解系统为何判定为幻觉。

验收标准：

- **AC-03.1** 每条成功结果包含 `is_hallucination`、`labels`、`primary_type`、`severity`、`review_required`、`claims`、`omissions`、`summary`、`engine`、`model_name`、`detector_version` 和 `config_hash`；正常结果的 `primary_type` 与 `severity` 为 `null`，而不是伪造默认值。
- **AC-03.2** 每个原子声明展示规范化声明、`system_reply` 原文引用及字符区间、证据判定和理由；声明原文引用必须由本地代码精确验证。`supported`/`contradicted` 展示知识库原文引用及字符区间，`unsupported` 显示“知识库中未找到支持证据”。
- **AC-03.3** 系统单独检查回答是否遗漏会实质改变用户判断的重要条件；每个遗漏项展示缺失事实、知识库原文引用、风险和理由。
- **AC-03.4** 所有证据字符区间必须由本地代码验证与 `knowledge_base` 精确一致，失败时整条结果记为结构错误，不展示不可验证的“证据”。
- **AC-03.5** 结果支持按成功/失败/待复核状态、类型和风险筛选。
- **AC-03.6** 总览展示幻觉率、类型分布、高/中/低风险分布和失败数；分母只包含成功结果，并同时展示成功覆盖率。

#### US-04：可选人工复审与结构化纠错

作为质量负责人，我希望按运行选择是否开启人工复审，并在检测完成后逐条确认或修改错误点，以便形成可追溯的人工反馈并用于后续改进。

验收标准：

- **AC-04.1** 每次检测前提供“启用人工复审”总开关，默认关闭；检测开始后开关锁定，改变设置必须新建运行。
- **AC-04.2** 关闭时不显示复审组件、不生成复审数据；评测和实验性建议分析仅使用另行加载的官方人工标注。
- **AC-04.3** 开启时每条成功结果显示 `未复核`、`人工确认正确` 或 `人工确认有误`，并显示“已复核成功结果数 / 成功结果总数”；失败记录不能被伪装成人工确认结果。
- **AC-04.4** 标记“检测正确”时保存人工确认，但不修改原始预测。
- **AC-04.5** 标记“检测有误”时，可修改是否幻觉、多标签、主类型、风险、总结、原子声明和关键遗漏；声明的回复原文引用及知识库证据引用仍必须通过本地字符区间校验。
- **AC-04.6** 用户可新增模型漏掉的声明或遗漏项、删除多检项，并恢复到模型原始结果。
- **AC-04.7** 每次保存生成新的人工修订版本，页面展示修改前后差异和 UTC 修订时间；原始预测不可覆盖。
- **AC-04.8** 未复核记录不得默认视为正确。
- **AC-04.9** 可导出带契约版本、`run_id`、输入哈希、原始预测哈希和连续修订事件链的人工反馈 JSON。MVP 不支持把反馈重新导入运行，也不支持进程重启后的运行恢复；反馈导入与跨会话恢复移至 v1.1。

#### US-05：评测漏检与误报

作为质量负责人，我希望在检测完成后加载人工标注，以便计算指标并明确知道漏检和误报了哪些案例。

验收标准：

- **AC-05.1** 用户显式冻结预测快照后，才能加载 `task4_ground_truth.json` 或同结构标注；冻结前评测入口不可用。
- **AC-05.2** 预测与标注按 ID 对齐；只评测成功结果与标注的交集，并列出预测独有、标注独有和失败 ID。
- **AC-05.3** 页面展示 TP、FP、TN、FN、混淆矩阵、幻觉 Precision、Recall、F1、正常回复召回率、Macro-F1、Balanced Accuracy、主类型匹配率和高风险召回率；分母为 0 时返回 `null` 并说明原因。
- **AC-05.4** 主类型匹配率的分母是“人工标注为幻觉且模型也预测为幻觉、双方类型均可映射”的记录；二分类漏检不进入该辅助指标分母，但必须在 FN 中体现。
- **AC-05.5** 高风险召回率的分母是风险参考中被标为高风险的人工幻觉记录，分子是其中预测为幻觉的成功记录；风险参考不完整时该指标返回 `null`。
- **AC-05.6** 页面分别列出漏检和误报的 ID、原文、预测理由与人工理由。
- **AC-05.7** Accuracy 只作为补充指标，不作为核心效果结论。
- **AC-05.8** `coverage = 成功且 ID 匹配数 / 人工标注数`；低于 100% 时所有指标标记为“不完整”，但仍可生成带明显局限性提示的实验性建议。

#### US-06：分析误判原因

作为质量负责人，我希望系统归纳每个漏检和误报的根因，以便知道检测器应该改进什么。

验收标准：

- **AC-06.1** 原因至少覆盖声明漏提取、证据误读、无依据边界错误、能力越界漏识别、部分支持误判、关键遗漏边界、一般表达误报和语义等价或否定关系误判。
- **AC-06.2** 每个错误案例展示至少一个主原因、可选次原因、依据和对应改进建议。
- **AC-06.3** 原因分析不得把样本 ID、完整问题或完整回复写入通用建议；对问题和回复执行 Unicode NFKC、转小写、合并空白后，建议不得连续复制任一来源文本 32 个或更多字符。
- **AC-06.4** LLM 分析失败时保留失败记录，不生成本地伪分析。

#### US-07：生成实验性改进建议

作为质量负责人，我希望系统根据漏检和误报生成受约束的实验性建议，以便理解可能的改进方向，同时避免把 20 条小样本上的变化误当成可部署升级。

验收标准：

- **AC-07.1** 系统可以通过真实 LLM 基于 FN/FP 归因生成提示词原则、判断边界或泛化示例建议；MVP 不生成或调整数值阈值。单次建议任务最多 8 次网络尝试、单响应最多 2,000 completion tokens，Provider 计量 `usage.total_tokens` 达到或超过 50,000 后停止，墙钟时限为 5 分钟。
- **AC-07.2** 建议只保存为结构化 JSON 和 Markdown，不写入只读检测器资源、不创建检测器版本、不修改活动指针，也不得生成或执行 Python、模板、文件操作或网络操作代码。
- **AC-07.3** 每条建议记录标签来源、输入数据哈希、原始预测哈希、实际模型名、基线检测器版本、生成时间、适用范围和“小样本实验性建议，不代表效果提升”的固定警告。
- **AC-07.4** 官方标注与人工修订是独立来源；同一 ID 冲突时必须展示冲突并要求用户为整个分析批次明确选择单一来源，禁止逐条静默混合。
- **AC-07.5** 选择人工修订时，复核覆盖率必须达到成功结果的 100%，且原始预测哈希全部匹配；否则只能导出已有反馈，不能调用建议生成。
- **AC-07.6** 建议生成请求不得包含样本 ID；完整问题和回复只作为明确分隔的不可信分析材料使用。对输出与来源文本执行 Unicode NFKC、转小写、合并空白后，通用建议不得连续复制任一来源文本 32 个或更多字符，也不得包含输入中出现的任何样本 ID。
- **AC-07.7** 页面和导出不得使用 `validated`、`improved`、`upgrade`、`out-of-fold` 或 `full-data replay` 描述这些建议。

#### US-08：导出交付报告

作为测评审阅者，我希望导出结构化结果和可读报告，以便提交、复查和截图展示。

验收标准：

- **AC-08.1** 可导出带契约版本的预测 JSON、评测 JSON、人工反馈 JSON 和实验性建议 JSON。
- **AC-08.2** 可导出 Markdown 综合报告。
- **AC-08.3** Markdown 报告包含分类定义、检测方法、覆盖率、指标及分母、失败项、漏检、误报、人工复审摘要、误判原因、实验性建议、局限性和 AI 工具使用情况。
- **AC-08.4** 报告只允许把实际冻结预测的评测标为 `baseline`；MVP 不产生 `out-of-fold`、`full-data replay` 或候选版本指标。
- **AC-08.5** 导出内容明确区分 `model_prediction`、`official_ground_truth`、`human_revision` 和 `experimental_suggestion`，并对用户文本使用安全的 Markdown 代码围栏或转义，禁止注入原始 HTML。

### Functional Classification Requirements

MVP 采用闭集知识假设：输入中的 `knowledge_base` 是当前任务的唯一可信业务事实来源。重要事实与知识库冲突，或没有知识库支持却以确定语气陈述时，属于幻觉候选。礼貌、同理心和不改变用户决策的一般表达不直接判为幻觉。

系统采用五类统一标签：

| 类型 | 业务定义 |
| --- | --- |
| 知识冲突 | 回复与知识库中的政策、优惠、参数、服务规则或事实明确矛盾 |
| 无依据编造 | 知识库没有支持信息，回复却确定陈述具体事实、地址、功能或品牌关系 |
| 能力越界 | 声称完成系统没有能力执行或查询的操作 |
| 安全误导 | 在健康、人身等高风险问题上违背证据或给出过度确定建议 |
| 关键遗漏或歪曲 | 省略会实质改变用户判断的重要条件，使整体结论失真 |

一条回复可包含多个标签，并必须选择一个主类型。主类型依次按风险更高、证据更明确、对核心问题影响更大确定。

风险采用动态三级制：

- 高：可能造成健康伤害、资金或权益损失、寄错地址，或虚假声称已执行关键操作；
- 中：可能影响购买决策、售后判断、兼容性或服务预期；
- 低：错误存在，但对核心决策和后续动作影响较小。

### Non-Goals

- 账号、权限、多租户和审计后台。
- 数据库、历史任务中心和分布式任务队列。
- 在线知识库检索、向量数据库和 RAG 管理。
- 模型微调、生产级监控和线上告警。
- CSV 支持、任意字段映射和客服系统写回。
- 自动生成、修改或执行程序代码。
- 自动生成可部署检测器、数值阈值搜索、双折交叉验证、候选激活、活动版本指针和版本回滚。
- 人工反馈重新导入、跨进程运行恢复和历史任务中心。
- 多人协同复审、审核任务分配和审批流。
- Streamlit、第二个 HTTP 服务、前端构建框架和运行时 CDN。
- 本地规则检测、规则降级、离线演示模式和运行时 mock LLM。
- 以当前 20 条小样本结果宣称生产泛化能力。

## 3. AI System Requirements

### Tool Requirements

- Python `>=3.11`。
- FastAPI `>=0.115,<1.0`，并作为唯一 HTTP 服务。
- Uvicorn `>=0.34,<1.0`。
- Pydantic `>=2.10,<3.0`。
- Jinja2 `>=3.1,<4.0`。
- HTMX 固定为 `2.0.10`，ECharts 固定为 `5.5.1`；两者必须作为本地静态资源提供，运行时不得依赖 CDN。
- pytest `>=8,<10`、Ruff `>=0.9,<1.0`、mypy `>=1.14,<2.0`、build `>=1.2,<2.0`。
- 所有 Python 依赖和工具配置统一写入根目录 `pyproject.toml`，不得维护第二套依赖来源。
- 强制使用可配置的真实 LLM API，密钥、基础地址和模型名通过环境变量提供。
- 运行时只注册 LLM Provider；抽象 Provider 接口仅用于隔离具体 SDK 和注入测试 mock。
- 检测采用证据优先的三阶段流程：提取原子声明、与知识库逐项核验、独立检查关键遗漏，最后聚合结果。
- 实验性建议由 LLM 生成，只输出提示词原则、标签边界和泛化示例建议；不得生成可激活配置或数值阈值。
- 人工复审关闭时，人工反馈模块不得创建空记录或把未复核结果解释为人工确认。
- 人工复审开启时，原始预测、官方标注和人工修订必须分别存储并携带来源标记。

### Evaluation Strategy

以“存在幻觉”为正类，必须报告：

- TP、FP、TN、FN；
- 幻觉 Precision、Recall、F1；
- 正常回复召回率；
- Macro-F1 和 Balanced Accuracy；
- 主类型匹配率；
- 高风险幻觉召回率。

原始人工标注没有严重程度字段。MVP 必须随应用交付一份经人工复核、版本化且只读的内置风险参考映射，其键只能覆盖内置人工标注中的幻觉正例。映射记录契约版本、标注集内容哈希、风险规则版本、每个正例的严重程度和文件内容哈希；不得由运行时代码生成或修改。若上传的人工标注对全部幻觉正例都提供合法 `severity`，则按该次标注内容哈希构建仅存在于当前运行评测对象中的上传风险参考；只提供部分严重程度时，高风险召回率为 `null`，不得混用内置映射补齐。

类型匹配使用预先声明、版本化的兼容关系。人工标注中的政策、优惠、参数和信息编造类标签，可兼容“知识冲突”或“无依据编造”；能力越界、安全误导、信息遗漏分别对应同名或语义等价的统一类型。类型兼容只影响辅助指标，不影响二分类指标。

MVP 不评估或宣称建议带来效果提升，不执行双折交叉验证、候选回放或模型选择。实验性建议只用于归纳误差与人工讨论；任何后续效果验证必须使用扩充且重新平衡的数据集、预先冻结的独立测试集和单独批准的后续设计。

## 4. Technical Specifications

### Architecture Overview

FastAPI 直接提供 JSON API、Jinja2 页面、HTMX 片段、JSON 文本上传、文件下载和本地静态资源，并且是项目唯一 HTTP 服务。产品内部包含 API/页面适配、数据校验、检测、推理适配、人工复审、评测、实验性建议和报告能力。路由、模板和浏览器脚本不得实现或复制业务规则；详细组件边界、数据契约、状态机、OpenAI-compatible Provider 协议和算法见关联设计文档。

### Integration Points

- 输入：`task4_replies.json` 或同结构 JSON 数组。
- 人工标注：`task4_ground_truth.json` 或同结构 JSON 数组。
- 外部依赖：符合技术设计所定义 Chat Completions 请求/响应契约、支持非流式结构化 JSON 输出的真实 LLM API；无有效配置时检测和实验性建议功能不可启动。
- 输出：预测 JSON、评测 JSON、人工反馈 JSON、实验性建议 JSON、Markdown 综合报告。
- HTTP：FastAPI 提供页面、HTMX 片段、JSON API、文件上传、下载和静态资源。
- 存储：MVP 不使用数据库；预测、评测、人工反馈、实验性建议和报告只能写入技术设计指定的 `runtime/` 输出目录。原始上传文件字节不持久化；评测产物可以保存计算所需的规范化标注字段、人工理由和来源哈希，但不得保存原始上传文件副本。MVP 不持久化可激活检测器版本。
- 认证：MVP 不提供账号认证。

### Standard Commands

- 安装开发依赖：`python -m pip install -e ".[dev]"`
- 运行测试：`python -m pytest -q`
- 代码检查：`python -m ruff check .`
- 格式检查：`python -m ruff format --check .`
- 类型检查：`python -m mypy src tests`
- 构建制品：`python -m build`
- 启动应用：`python -m uvicorn src.api.app:app --reload`

### Security & Privacy

- API 密钥不得写入源码、日志、页面、异常详情、报告、截图、测试夹具或导出文件。
- 服务仅支持一个应用 worker 和 loopback 访问；开发命令的 reload supervisor 可以存在，但不得启动多个应用 worker 或共享 `runtime/`。不得作为公网或局域网共享服务部署。状态修改请求必须执行同源检查，不配置 CORS 放行。
- 人工标注不得进入首次检测请求。
- 内置风险参考不得进入检测模块依赖、检测提示词或聚合输入。
- 人工修订不得覆盖原始预测，所有修订必须关联原始预测哈希。
- 上传内容默认只存在当前进程；原始上传文件不得写入运行时目录或其他存储，也不得上传到 LLM 之外的服务。
- 用户必须在每次运行前确认待检测内容和实验性建议分析材料会发送到所配置的外部 LLM；产品必须提示外部服务的数据保留、地域和合规策略不由本应用控制。
- Provider 必须返回 token usage 才能执行检测；应用以次数、Provider usage 和墙钟时间限制风险，但无法保证第三方实际计费金额，页面不得展示虚构费用估算。
- 所有进入提示词的用户文本和知识库必须作为有明确边界的不可信数据，模型不得服从其中的指令；所有模型输出仍需本地白名单与聚合不变量校验。
- Jinja2 必须开启自动转义，模板不得对用户或模型文本使用 `safe`；浏览器脚本使用 `textContent` 或等价安全 API；Markdown 导出不得保留用户提供的原始 HTML。
- 实验性建议不得生成或执行代码，不得写入只读检测器配置。
- 日志只记录必要诊断元数据，不记录完整外部模型响应。

### Phased Rollout

#### MVP：测评交付

- FastAPI + Jinja2 + HTMX + ECharts Dashboard；

### Technical Risks

| 风险 | 影响 | MVP 应对 |
| --- | --- | --- |
| 仅 20 条样例且只有 2 个正常样本 | 指标方差高，易过拟合 | 报告类别指标、覆盖率和明确局限性；MVP 不验证或激活改进建议 |
| 知识库可能不完整 | “无依据”不一定是客观错误 | 明确闭集假设，并保留人工复核标记 |
| LLM 输出不稳定 | 结构错误或重复运行差异 | 低随机性、结构校验、一次 LLM 修复；仍失败则记录失败并允许重试 |
| 输入过大或声明过多 | 超出上下文、费用失控或本地拒绝服务 | 固定请求体、记录数、字符数和声明数上限；启动前显示调用上界；单应用 worker 只运行一个外部任务 |
| 长任务无法停止或 token 费用失控 | 任务占用唯一 worker 数小时或产生超预期调用 | 30/5 分钟任务时限、用户协作式取消、200 次检测请求上限、250,000 total tokens 断路值、单响应 2,000 completion tokens；明确断路值可被最后一次响应突破且不承诺货币金额 |
| 本地前端资源缺失或版本漂移 | 页面交互或图表无法离线运行 | 固定 HTMX/ECharts 版本、随应用提供静态文件并测试静态路由 |
| 关键遗漏边界主观 | 标注与模型可能分歧 | 输出证据和理由，类型指标仅作辅助 |
| 实验性建议发生答案记忆或被误解为升级 | 建议只复述样本，或被当成已验证改进 | 禁止 ID/完整原文绑定、固定实验性警告、不生成版本、不执行验证、不提供激活与回滚 |
| 人工复审不完整或来源冲突 | 把未复核样本当正确，或覆盖官方标注 | 展示覆盖率、三类数据分离、冲突显式选择；覆盖不足时禁止以人工修订生成建议 |
| API 限流或不可用 | 批次不完整且无法离线检测 | 最多重试 2 次、单条失败隔离、覆盖率标记和人工重试；禁止本地降级 |
| Prompt Injection 或展示层注入 | 模型分类被不可信文本影响，或页面/报告执行恶意内容 | 明确数据边界、结构白名单、证据区间校验、Jinja 自动转义、禁止 `safe`/`innerHTML`/原始 HTML |
| 客服内容含个人或敏感信息 | 内容被发送到外部 LLM 并受其保留政策约束 | 每次运行显式确认、页面提示责任边界、只发送当前记录、应用不额外持久化原始上传内容 |
