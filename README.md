# 客服回复幻觉检测

## 1. 分类体系

系统将客服回复中的风险分为五类：

| 类型 | 定义 | 严重程度 |
|------|------|----------|
| **知识冲突** | 回复与知识库中的政策、参数或事实相矛盾 | 高/中 |
| **无依据编造** | 知识库没有依据，却给出确定事实、优惠或结论 | 高/中 |
| **能力越界** | 系统没有对应接口或权限，却声称已经查询、修改、升级或执行操作 | 高 |
| **安全误导** | 涉及健康、资金、权益或安全时给出可能造成实际损害的错误建议 | 高 |
| **关键遗漏或歪曲** | 遗漏会改变用户决策的重要条件，或只保留部分事实导致错误理解 | 高/中/低 |

**严重程度定义**：
- **高**：可能造成健康、资金、权益损失或虚假执行关键操作
- **中**：会明显误导业务判断
- **低**：影响有限但仍缺少可靠依据

一个样本可以有多个标签，主类型按风险优先级确定。

## 2. 指标定义及理由

### 核心指标

| 指标 | 定义 | 理由 |
|------|------|------|
| **Precision（精确率）** | TP / (TP + FP) | 衡量检出结果中真正有幻觉的比例，避免误报干扰客服人员 |
| **Recall（召回率）** | TP / (TP + FN) | 衡量实际幻觉被检出的比例，避免漏检导致用户受损 |
| **F1 Score** | 2 × Precision × Recall / (Precision + Recall) | 精确率和召回率的调和平均，综合评估检测性能 |
| **TP（真阳性）** | 正确识别为幻觉的记录数 | 检测有效的直接证据 |
| **FP（假阳性）** | 错误识别为幻觉的记录数 | 衡量误报程度 |
| **FN（假阴性）** | 未识别出的幻觉记录数 | 衡量漏检程度 |

### 指标选择理由

1. **Recall 优先**：客服幻觉可能导致用户资金损失或健康风险，漏检代价高于误报
2. **Precision 兼顾**：过多误报会增加人工复审成本，降低系统可用性
3. **F1 综合评估**：在 Recall 和 Precision 之间取得平衡

## 3. 评估方法

### 3.1 检测流程

```
客服回复 → 声明抽取 → 证据判断 → 完整性检查 → 分类聚合 → 结果输出
```

1. **声明抽取**：从 system reply 中提取可验证的事实性声明
2. **证据判断**：将每个声明与 knowledge base 对比，判断支持/矛盾/无依据
3. **完整性检查**：检测关键信息遗漏
4. **分类聚合**：根据判断结果生成最终分类

### 3.2 评测方法

1. **冻结预测**：检测完成后冻结结果，防止后续修改
2. **独立标注**：使用 `task4_ground_truth.json` 中的人工标注作为标准
3. **逐条对比**：将预测结果与标注逐条对比，计算 TP/FP/FN/TN
4. **指标计算**：基于混淆矩阵计算 Precision、Recall、F1

### 3.3 人工复审

- 支持"确认正确"和"修正"两种复审操作
- 复审结果独立存储，不覆盖原始预测
- 支持基于复审结果生成改进建议

### 误报分析

| 误报记录 | 原因分析 |
|----------|----------|
| h12 | 系统回复与知识库一致，但检测边界过宽，将正常表达判为幻觉 |

**改进建议**：仅在存在知识冲突、缺少必要依据或关键事实遗漏时输出阳性标签。

## 5. 检测方法

### 技术架构

- **后端**：Python 3.11+ / FastAPI
- **前端**：HTML + HTMX + ECharts
- **LLM**：OpenAI Chat Completions 兼容接口（支持 DeepSeek 等）
- **并发**：ThreadPoolExecutor 多线程并行检测（默认5线程）

### 检测流程

```
POST /runs → 创建运行
    ↓
并行检测（5线程） → 进度监控
    ↓
冻结结果 → 上传 Ground Truth → 运行评测
    ↓
人工复审 → 生成建议 → 导出结果
```

### 快速开始

```powershell
# 设置环境变量
$env:HALLUCINATION_API_KEY="your-api-key"
$env:HALLUCINATION_BASE_URL="https://api.deepseek.com"
$env:HALLUCINATION_MODEL="deepseek-v4-flash"

# 启动应用
python -m uvicorn src.api.app:app --host 127.0.0.1 --port 8000 --reload

# 访问 http://127.0.0.1:8000/
```

### Mock 模式（无需 API）

```powershell
$env:HALLUCINATION_API_KEY="mock-key"
$env:HALLUCINATION_BASE_URL="http://127.0.0.1:9"
$env:HALLUCINATION_MODEL="mock"
python -m uvicorn src.api.app:app --reload
```

## 8. 局限性讨论

### 数据局限

1. **样本规模小**：仅 20 条回复，不能代表真实业务场景
2. **类别不平衡**：阳性样本占多数，缺乏足够的阴性样本验证
3. **领域单一**：仅覆盖电商客服场景，其他领域需重新验证

### 模型局限

1. **LLM 依赖**：检测质量依赖底层模型能力
2. **结构输出风险**：即使通过 JSON 校验，也可能存在语义边界错误
3. **上下文限制**：长文本可能超出模型上下文窗口

### 方法局限

1. **检测边界**：非事实性表达（如礼貌用语）可能被误判
2. **知识库依赖**：需要完整的知识库作为对比基准
3. **实时性**：每次检测需要调用外部 API，存在延迟

### 改进方向

1. **扩大样本**：收集更多真实业务数据，覆盖更多场景
2. **优化边界**：引入更精确的事实性判断逻辑
3. **缓存机制**：对相似查询结果进行缓存，减少 API 调用
4. **人机协同**：将检测结果作为辅助，结合人工复审
