# 项目开发约束

## 适用范围与需求依据

本文件适用于仓库根目录及全部子目录。

- 10小时之内必须完成
- 产品范围和验收标准以 `docs/requirements/2026-07-22-0110-customer-service-hallucination-detection-prd.md` 为准。
- 领域模型和算法设计以 `docs/superpowers/specs/2026-07-22-0110-customer-service-hallucination-detection-design.md` 为准。
- 不得擅自扩展 MVP 范围。数据库、认证、多租户、分布式任务队列、生产监控、在线知识库集成和自动生成代码均不属于当前范围。

## 技术栈与最低版本

- Python `>=3.11`。
- FastAPI `>=0.115,<1.0`，并作为项目唯一 HTTP 服务。
- Uvicorn `>=0.34,<1.0`。
- Pydantic `>=2.10,<3.0`。
- Jinja2 `>=3.1,<4.0`，用于服务端页面模板。
- HTMX `2.x`，用于页面局部更新和交互。
- ECharts `5.x`，用于图表展示。
- pytest `>=8,<10`。
- Ruff `>=0.9,<1.0`。
- mypy `>=1.14,<2.0`。
- build `>=1.2,<2.0`。
- HTMX 和 ECharts 必须固定版本并作为本地静态资源提供，运行时不得依赖 CDN。
- 所有 Python 依赖和工具配置统一写入根目录的 `pyproject.toml`，不得同时维护多套依赖来源。

## 标准命令

所有命令从仓库根目录运行：

```text
安装开发依赖：python -m pip install -e ".[dev]"
运行测试：    python -m pytest -q
代码检查：    python -m ruff check .
格式检查：    python -m ruff format --check .
类型检查：    python -m mypy src tests
构建制品：    python -m build
启动应用：    python -m uvicorn src.api.app:app --reload
```

- 宣布任务完成前，必须重新运行与本次修改相关的检查。
- 合并或交付前，必须运行完整的测试、lint、格式、类型检查和 build 命令，并报告最新结果。
- 如果项目尚未创建，导致某条命令无法运行，应明确说明“尚不可运行”，不得声称检查已经通过。
- 在进行git提交前必需经过用户同意

## HTTP 与前端架构

- FastAPI 直接处理 JSON API、Jinja2 页面、HTMX 片段、文件上传、文件下载和静态资源请求。
- 不得引入 Streamlit 或第二个 HTTP 服务。
- `src/api/` 只负责路由、请求校验、响应转换、模板渲染、静态文件挂载和依赖注入。
- HTTP 路由和模板不得包含检测、评测、自学习等业务算法。
- 前端页面通过 FastAPI 提供的应用服务完成操作，不得在 JavaScript 或模板中复制业务规则。

## 模块边界

- 领域模型不得依赖 FastAPI、Jinja2、HTMX、ECharts 或具体 LLM SDK。
- 检测模块只能依赖抽象推理 Provider，不得读取官方人工标注或人工修订。
- 评测模块只接收冻结预测和官方人工标注，不得调用或修改检测器。
- 人工复审模块只能基于冻结预测追加确认和修订，不得覆盖原始预测或历史修订。 。
- 版本模块只接受结构校验通过的声明式配置。
- 升级门禁必须是纯判断逻辑；回滚只能修改活动版本指针，不得删除历史版本。
- `model_prediction`、`official_ground_truth` 和 `human_revision` 必须在类型、模块、存储、页面和导出结果中保持隔离。
- 必须保持输入顺序稳定，并记录固定随机种子、版本元数据、输入哈希、重试和降级信息，保证结果可复现。

## 安全与数据处理

- LLM 配置只能从以下环境变量读取：
  - `HALLUCINATION_API_KEY`
  - `HALLUCINATION_BASE_URL`
  - `HALLUCINATION_MODEL`
- 密钥不得写入源码、日志、页面状态、异常详情、报告、截图、测试夹具或导出文件。
- 首次检测不得接触官方人工标注或人工修订。
- 双折交叉验证中，候选生成阶段不得接触验证折答案。
- 上传内容默认只存在于当前进程；除非用户主动导出，不得写入其他存储或上传到其他服务。
- 日志只记录必要的诊断元数据，不得记录完整的外部模型响应。
- 单条检测失败必须保留错误记录，并明确排除在指标分母之外；不得通过跳过失败记录提高指标。

## 禁止修改的内容

未经用户明确批准，不得修改：

- `task4_replies.json`；
- `task4_ground_truth.json`；
- `docs/requirements/**`；
- `docs/superpowers/specs/**` 下已有的已批准文档；
- `.git/**`；
- `.agents/**`；
- IDE 配置目录。

运行时代码不得修改源码、测试、基准输入或人工标注。预测、评测、人工反馈、报告和检测器版本只能写入设计指定的运行时输出目录。

## 依赖管理

- 技术栈章节列出的依赖已经批准。
- 新增其他运行时依赖前必须获得用户确认。
- 新增开发依赖时必须说明用途，不得为了单一小功能引入大型工具。
- 不得在未确认的情况下升级依赖主版本。
- 不得引入 Streamlit、数据库 ORM、任务队列或前端构建框架。

## 兼容性、部署与回滚

- MVP 不使用数据库，因此不包含数据库迁移。
- 现有输入和输出 JSON 契约必须保持向后兼容。
- 破坏性字段变更必须引入新版本，并在实施前获得用户确认。
- 检测器配置和类型兼容表必须版本化，不得根据单次评测结果静默调整。
- 回滚只允许切换活动版本指针，不得覆盖或删除历史版本。
- 未经确认，不得部署、发布、push、创建 Pull Request、调用真实外部 LLM、激活候选检测器或执行版本回滚。

## 必须先确认的操作

执行以下操作前必须获得用户明确确认：

- 新增未批准的运行时依赖；
- 修改 PRD、已批准设计、基准输入或官方人工标注；
- 将样本数据发送到真实外部 LLM，或发起可能产生费用的 API 调用；
- 修改公开数据契约；
- 放宽预测隔离、交叉验证隔离或升级门禁；
- 删除、覆盖、迁移或批量重写文件；
- 写入工作区之外的位置；
- push、创建 Pull Request、部署或发布；
- 激活候选检测器版本或执行回滚。

只读检查和本地测试、lint、格式检查、类型检查、build 不需要额外确认。

## TDD 规则

以下功能必须严格执行 RED-GREEN-REFACTOR：

- 领域规则和聚合不变量；
- FastAPI 路由及请求、响应契约；
- Provider 契约、重试、结构修复和降级逻辑；
- 评测指标、数据隔离、交叉验证和升级门禁；
- 人工修订、版本激活和回滚；
- 所有 bug 修复，必须先增加能够复现问题的失败测试。

允许不先写测试的情况仅限：

- 纯文档修改；
- 不改变行为的格式化；
- 静态 HTML/CSS 视觉探索；
- 不包含可执行行为的声明式示例配置；
- 演示截图。

探索性页面或配置一旦成为正式交付的交互、校验规则或业务行为，必须补齐测试并执行完整的 RED-GREEN-REFACTOR。

## 实施纪律

- 遵循已批准的模块边界和现有命名，不做与当前目标无关的重构。
- 文件应保持单一职责，模块之间使用明确接口。
- 不得在 HTTP 路由、模板和领域服务中重复业务规则。
- 当前 20 条基准数据规模小且类别不平衡；必须报告分类指标和局限性，不得据此宣称具备生产泛化能力。
- 遇到失败时先调查根因，不得连续叠加猜测性修改。
- 完成声明必须基于最新命令输出和逐项需求检查，不能仅依据 Agent 的完成报告。
